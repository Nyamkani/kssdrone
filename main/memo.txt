# 1. 현재 구조의 핵심 변화

처음에는 `Armed()` 안에서 사실상 이런 구조였습니다.

```text id="2rm6xk"
Armed()
  prepare 전부
  slow comp 전부
  PID 전부
  output 전부
```

즉 매 loop마다 너무 많은 일을 했습니다.

지금은 이렇게 바뀐 상태입니다.

```text id="tmgjxf"
Armed()
  2kHz급:
    IMU
    PID

  1kHz:
    DShot output

  500Hz:
    command
    EKF
    failsafe
    tilt safety

  250Hz 이하:
    airborne
    soft comp
    output comp
    log
```

즉 “시간에 따른 함수 실행”이라기보다 정확히는:

```text id="jjm525"
각 함수마다 원하는 실행 주기를 가지고,
dt_accum_으로 실행 시점을 결정하는 구조
```

입니다.

예:

```cpp id="4p4k6h"
float armed_command_dt_accum_ = 0.0f;
float armed_estimator_dt_accum_ = 0.0f;
float armed_motor_output_dt_accum_ = 0.0f;
```

이 변수들은 **데이터 저장용 cache가 아닙니다.**

얘네는 단순히:

```text id="8gdl8j"
"이 함수가 마지막으로 실행된 뒤 시간이 얼마나 지났는가?"
```

를 기억하는 **스케줄러용 시간 누적값**입니다.

---

# 2. `cache`와 `ArmedContext ctx{}`의 차이

이게 제일 중요합니다.

## 짧게 말하면

```text id="dfzeox"
멤버 cache:
  loop와 loop 사이에 살아남는 값

ArmedContext ctx:
  이번 Armed() 한 번 안에서만 쓰는 임시 작업 공간
```

입니다.

---

## 멤버 cache

예를 들어 이런 값들입니다.

```cpp id="lp35pz"
AttitudeTarget armed_target_cache_{};
bool armed_target_valid_ = false;

IMU_PARESED_DATA armed_imu_cache_{};
bool armed_imu_valid_ = false;

EKFAttitudeOutput armed_ekf_out_cache_{};
AttitudeState armed_state_cache_{};
bool armed_state_valid_ = false;

FlightPIDOutput armed_pid_out_cache_{};
bool armed_pid_valid_ = false;
```

이 값들은 `KSSDrone` 객체의 멤버입니다.
따라서 `Armed()`가 끝나도 사라지지 않습니다.

즉 다음 loop에서도 남아 있습니다.

```text id="7snzhm"
loop 100에서 계산한 target
→ loop 101, 102에서도 사용 가능

loop 100에서 계산한 EKF state
→ estimator가 쉬는 loop 101에서도 사용 가능
```

이게 cache의 역할입니다.

---

## `ArmedContext ctx{}`

반면 이건 `Armed()` 함수 안에서 매번 새로 만들어지는 지역 변수입니다.

```cpp id="diwhd4"
esp_err_t KSSDrone::Armed(const float dt)
{
    ArmedContext ctx{};
    this->LoadArmedContextCache(ctx);

    ...
}
```

`ctx`는 이번 loop 안에서만 존재합니다.

```text id="h4auqk"
Armed() 진입
  ctx 생성
  cache에서 현재값 복사
  이번 loop 함수들이 ctx를 읽고/수정
Armed() 종료
  ctx 사라짐
```

즉 `ctx`는 **영구 저장소가 아닙니다.**

---

# 3. 왜 둘 다 필요한가?

subrate 구조에서는 어떤 함수는 매 loop 돌고, 어떤 함수는 가끔만 돕니다.

예를 들어:

```text id="pm5748"
PID:      2kHz
Command: 500Hz
EKF:     500Hz
DShot:   1kHz
```

그러면 2kHz PID는 매 loop마다 target과 attitude state가 필요합니다.
하지만 target은 command 함수가 500Hz로만 갱신합니다.

따라서 command 함수가 안 도는 loop에서도 PID가 target을 써야 합니다.

이때 필요한 것이 cache입니다.

```text id="5ixb06"
CommandMedium 실행됨:
  새 target 계산
  ctx.target 갱신
  armed_target_cache_ 갱신

CommandMedium 실행 안 됨:
  LoadArmedContextCache()가
  armed_target_cache_를 ctx.target에 복사
```

그래서 PID는 항상 이렇게 볼 수 있습니다.

```cpp id="n4riit"
if (!ctx.target_valid || !ctx.state_valid || !ctx.imu_valid)
{
    return ESP_OK;
}
```

즉 `ctx` 입장에서는 command가 이번 loop에 돌았는지 아닌지 몰라도 됩니다.
항상 “현재 사용할 수 있는 최신값”이 들어있게 만드는 구조입니다.

# 4. 비유하면

```text id="mzwfmp"
멤버 cache:
  전역 게시판 / 저장소

ArmedContext ctx:
  이번 loop에서 쓰려고 책상 위에 복사해 둔 작업지

dt_accum_:
  각 작업을 언제 할지 알려주는 타이머
```

입니다.

# 5. 데이터 흐름 예시

## Command target

```text id="wbrqbu"
ArmedCommandMedium() 500Hz
  active_cmd_ 읽음
  target 계산
  ctx.target 갱신
  armed_target_cache_ 갱신
```

왜 cache가 필요한가?

```text id="04rpsy"
PID는 2kHz인데 command는 500Hz라서,
command가 쉬는 3개의 loop에서도 target이 필요함
```

---

## IMU

```text id="mddazk"
ArmedImuFast() 2kHz
  IMU 직접 read
  LPF 적용
  ctx.imu 갱신
  armed_imu_cache_ 갱신
```

IMU는 거의 매 loop 새로 읽기 때문에 cache 중요도가 낮아 보일 수 있습니다.
그래도 estimator나 output이 skip되는 loop, 또는 순서상 다른 함수가 최신 IMU를 써야 하므로 cache를 유지하는 게 좋습니다.

---

## EKF / state

```text id="rq85mz"
ArmedEstimatorMedium() 500Hz
  ctx.imu 사용
  EKF predict/update
  attitude state 계산
  ctx.state 갱신
  armed_state_cache_ 갱신
```

EKF가 안 도는 loop에서는:

```text id="p9me4y"
LoadArmedContextCache()
  이전 EKF state를 ctx.state로 복사
```

그리고 PID 직전에 gyro만 최신 IMU로 덮어씁니다.

```cpp id="vnv49h"
ctx.state.gyro_x_rad_s = ctx.imu.gx_rad_s;
ctx.state.gyro_y_rad_s = ctx.imu.gy_rad_s;
ctx.state.gyro_z_rad_s = ctx.imu.gz_rad_s;
```

이 구조가 중요합니다.

```text id="n86xzj"
roll/pitch angle:
  500Hz EKF 값 사용

gyro rate:
  2kHz 최신 IMU 값 사용
```

그래서 EKF를 500Hz로 낮춰도 rate PID는 최신 gyro를 봅니다.

---

## PID output

```text id="ocnlig"
ArmedControlFast() 2kHz
  ctx.target
  ctx.state
  ctx.imu
  사용해서 PID 계산

  ctx.pid_out 갱신
  armed_pid_out_cache_ 갱신
```

왜 PID cache가 필요한가?

DShot output은 1kHz입니다.
즉 PID는 2번 계산되고, motor output은 그중 최신 PID 결과를 사용합니다.

```text id="frit3b"
loop A:
  PID 계산
  DShot 출력

loop B:
  PID 계산
  DShot 출력 안 함

loop C:
  PID 계산
  DShot 출력
```

DShot 출력 함수는 항상 최신 PID 결과를 쓰면 됩니다.

```cpp id="mxdnhw"
ctx.pid_out = this->armed_pid_out_cache_;
this->ArmedOutputFast(motor_dt, ctx);
```

# 6. cache와 ctx의 규칙

이 규칙으로 보면 됩니다.

## 규칙 1

**다음 loop에서도 필요한 값은 멤버 cache에 저장한다.**

예:

```text id="l3h2sq"
target
imu
ekf output
attitude state
pid output
throttle_rate_cmd
```

---

## 규칙 2

**이번 loop 안에서 뒤쪽 함수가 바로 써야 하는 값은 ctx에도 넣는다.**

예:

```cpp id="8orhgx"
ctx.target = target;
ctx.target_valid = true;

this->armed_target_cache_ = target;
this->armed_target_valid_ = true;
```

둘 다 하는 이유는:

```text id="c59qxv"
ctx:
  이번 loop의 뒤 함수가 바로 사용

cache:
  다음 loop들이 사용
```

입니다.

---

## 규칙 3

**ctx에만 넣고 cache에 안 넣으면, 다음 loop에서 사라진다.**

예를 들어 command 함수에서 이렇게만 하면:

```cpp id="risljs"
ctx.target = target;
ctx.target_valid = true;
```

이번 loop에서는 잘 됩니다.
하지만 다음 loop에서 command가 안 돌면 target이 없어집니다.

따라서 subrate output은 보통 cache도 같이 갱신해야 합니다.

---

## 규칙 4

**cache에만 넣고 ctx에 안 넣으면, 이번 loop 뒤 함수가 최신값을 못 본다.**

예:

```cpp id="6d4f3m"
this->armed_target_cache_ = target;
this->armed_target_valid_ = true;
```

이렇게만 하면 다음 loop에서는 보이지만, 이번 loop의 PID는 아직 예전 `ctx.target`을 볼 수 있습니다.

그래서 보통은 둘 다 합니다.

```cpp id="pym7po"
ctx.target = target;
ctx.target_valid = true;

this->armed_target_cache_ = target;
this->armed_target_valid_ = true;
```

---

# 7. 현재 구조를 표로 보면

| 데이터               |                          생성 함수 |     실행 주기 |  cache 필요 |                ctx 필요 |
| ----------------- | -----------------------------: | --------: | --------: | --------------------: |
| command           | ESP-NOW callback / active_cmd_ |     외부 수신 |     이미 멤버 |               포인터로 참조 |
| target            |         `ArmedCommandMedium()` | 250~500Hz |        필요 |                    필요 |
| throttle_rate_cmd |         `ArmedCommandMedium()` | 250~500Hz |        필요 |                    필요 |
| imu               |               `ArmedImuFast()` |      2kHz |    있으면 좋음 |                    필요 |
| ekf_out           |       `ArmedEstimatorMedium()` |     500Hz |        필요 |                    필요 |
| state angle       |       `ArmedEstimatorMedium()` |     500Hz |        필요 |                    필요 |
| latest gyro       |               `ArmedImuFast()` |      2kHz | imu cache | PID 직전 ctx.state에 덮어씀 |
| pid_out           |           `ArmedControlFast()` |      2kHz |        필요 |                    필요 |
| motor dshot       |            `ArmedOutputFast()` |      1kHz |        선택 |          output 직전 사용 |

# 8. `LoadArmedContextCache(ctx)`의 의미

이 함수는 매 loop 시작 시점에 이렇게 하는 역할입니다.

```text id="gjbjvg"
"지난 loop들에서 만들어진 최신 cache를
 이번 loop의 ctx에 깔아준다"
```

예:

```cpp id="wsai67"
void KSSDrone::LoadArmedContextCache(ArmedContext& ctx)
{
    if (this->armed_target_valid_)
    {
        ctx.target = this->armed_target_cache_;
        ctx.target_valid = true;
    }

    if (this->armed_imu_valid_)
    {
        ctx.imu = this->armed_imu_cache_;
        ctx.imu_valid = true;
    }

    if (this->armed_state_valid_)
    {
        ctx.state = this->armed_state_cache_;
        ctx.state_valid = true;
    }

    if (this->armed_pid_valid_)
    {
        ctx.pid_out = this->armed_pid_out_cache_;
    }

    ctx.throttle_rate_cmd = this->armed_throttle_rate_cmd_;
}
```

이게 없으면 subrate 구조가 깨집니다.

왜냐하면 command/EKF/output comp가 안 도는 loop에서는 `ctx`가 비어 있기 때문입니다.

# 9. `Armed()`의 현재 이상적인 흐름

개념적으로는 이렇게 됩니다.

```cpp id="978pe4"
esp_err_t KSSDrone::Armed(const float dt)
{
    ArmedContext ctx{};
    this->LoadArmedContextCache(ctx);

    /*
     * 2kHz
     */
    this->ArmedImuFast(dt, ctx);

    /*
     * 500Hz / 250Hz 계열
     */
    this->ArmedFailsafeMedium(dt, ctx);
    this->ArmedCommandMedium(dt, ctx);
    this->ArmedEstimatorMedium(dt, ctx);
    this->ArmedTiltSafetyMedium(dt, ctx);
    this->ArmedAirborneSlow(dt, ctx);
    this->ArmedSlowCompUpdate(dt, ctx);
    this->ArmedOutputCompMedium(dt, ctx);

    /*
     * 2kHz PID
     */
    if (RunControlPeriod(dt))
    {
        this->ArmedControlFast(control_dt, ctx);
    }

    /*
     * 1kHz motor output
     */
    if (RunMotorPeriod(dt))
    {
        ctx.pid_out = this->armed_pid_out_cache_;
        this->ArmedOutputFast(motor_dt, ctx);
    }

    return ESP_OK;
}
```

실제 순서는 약간 조정 가능합니다.
하지만 원칙은 이겁니다.

```text id="yh3stv"
1. ctx는 cache로 초기화
2. 이번 loop에서 새로 계산한 값은 ctx와 cache 둘 다 갱신
3. 뒤 함수는 ctx를 사용
4. 다음 loop는 cache를 사용
```

# 10. 흔한 실수

## 실수 1: ctx를 저장하려고 함

하면 안 됩니다.

```cpp id="zctjzk"
// 나쁜 예
this->ctx_ptr_ = &ctx;
```

`ctx`는 `Armed()`가 끝나면 사라지는 stack 변수입니다.

---

## 실수 2: valid flag 없이 값 사용

항상 이런 체크가 필요합니다.

```cpp id="i00vca"
if (!ctx.target_valid || !ctx.state_valid || !ctx.imu_valid)
{
    return ESP_OK;
}
```

subrate 구조에서는 “이번 loop에 그 함수가 안 돌았다”가 정상 상황입니다.
따라서 valid flag가 중요합니다.

---

## 실수 3: cache만 갱신하고 ctx를 안 갱신

이번 loop에서 뒤쪽 함수가 최신값을 못 봅니다.

---

## 실수 4: ctx만 갱신하고 cache를 안 갱신

다음 loop에서 값이 사라집니다.
